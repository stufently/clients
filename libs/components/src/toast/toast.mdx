# Toast

Toasts are brief, non-blocking notifications that communicate the result of an action. They appear
at the edge of the screen, auto-dismiss after a short delay, and require no user interaction to
close.

Use toasts for:

- Confirming a completed action ("Item saved", "Copied to clipboard")
- Surfacing a recoverable error that doesn't block the user
- Communicating background progress or status changes

**Avoid** using toasts for critical errors that require acknowledgment, multi-step flows, or content
that needs to persist.

---

## Variants

Four variants communicate severity and tone.

<Canvas of={stories.Variants} />

| Variant   | When to use                                       |
| --------- | ------------------------------------------------- |
| `success` | Action completed successfully                     |
| `error`   | Something went wrong; user may need to act        |
| `warning` | Potentially unintended outcome; proceed with care |
| `info`    | Neutral status update or background information   |

---

## Long content

Keep toast messages short and prefer a single `string`. The timeout auto-extends based on word count
so the user has time to read.

Passing an array of strings to render multiple paragraphs is **deprecated** and will be removed in a
future release. Use a single string instead.

## Usage

Toasts are triggered programmatically via `ToastService`. Inject it anywhere in an Angular component
or service.

```ts
import { ToastService } from "@bitwarden/components";
```

```ts
constructor(private toastService: ToastService) {}

this.toastService.showToast({
  variant: "success",
  message: "Your changes have been encrypted and saved.",
});
```

<Canvas of={stories.Service} />

### `showToast` options

| Option    | Type                                          | Required | Description                                      |
| --------- | --------------------------------------------- | -------- | ------------------------------------------------ |
| `message` | `string`                                      | Yes      | Toast body.                                      |
| `variant` | `"success" \| "error" \| "info" \| "warning"` | No       | Visual style. Defaults to `"info"`.              |
| `timeout` | `number`                                      | No       | Override the auto-dismiss delay in milliseconds. |

## Setup

`<bit-toast-container>` is already included in the layout components (`bit-layout`,
`bit-landing-layout`, `popup-tab-navigation`). No manual placement is needed in most apps.

### Positioning

The container is responsive: on narrow screens toasts appear full-width at the bottom; on wider
screens (≥ `sm` breakpoint) they anchor to the bottom-right corner.

### Container inputs

All inputs are optional and fall back to sensible defaults.

| Input          | Type     | Default | Description                                                                                     |
| -------------- | -------- | ------- | ----------------------------------------------------------------------------------------------- |
| `bottomOffset` | `string` | `1rem`  | Bottom offset. Use to clear fixed bottom chrome (e.g. a tab bar). Accepts any CSS length.       |
| `endOffset`    | `string` | `1rem`  | Inline-end offset on wide screens. Use to clear a side panel or drawer. Accepts any CSS length. |

---

## Behavior

### Auto-dismiss

Toasts dismiss automatically after `timeout` milliseconds. The timeout is calculated per toast — if
no explicit `timeout` is passed to `showToast`, the duration extends slightly for longer messages
(+1 second per 120 words beyond the baseline).

### Hover to pause

Moving the cursor over the toast area pauses all active timers. Timers resume from where they left
off when the cursor leaves.

### Stacking

When multiple toasts are active, they stack on top of each other — only the topmost toast is fully
visible. Each toast's timer only runs while it is on top; earlier toasts resume counting down once
they reach the top.

## Accessibility

- The container renders with `role="status"` and `aria-live="polite"`, so new toasts are announced
  by screen readers without interrupting the user.
- Error toasts additionally carry `role="alert"`, which triggers an assertive announcement and is
  appropriate for time-sensitive failures.
- The close button is always visible and keyboard-focusable.
- Toasts do not trap focus — they are supplementary and non-blocking.
